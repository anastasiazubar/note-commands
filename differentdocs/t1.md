## Test 1 – Read-Only Group Permission Validation

### Objective

Ensure that groups designated as read-only do not have permissions that allow resource creation, modification, deletion, or administrative actions.

### Validation

1. Identify groups following the read-only naming convention (e.g., *_RO).

2. Retrieve all IAM role bindings assigned to the group.

3. For each assigned role:

   * Identify whether the role is predefined or custom.
   * Extract all permissions included in the role.

4. Analyze the effective permissions granted to the group.

5. Compare permissions against a restricted permission list containing actions such as:

   * create
   * update
   * delete
   * modify
   * setIamPolicy
   * add
   * remove
   * admin
   * write

6. Flag any role that grants write or administrative capabilities regardless of whether it is a predefined or custom role.

### Examples

**Pass**

Role: roles/viewer

Permissions:

* resourcemanager.projects.get
* compute.instances.get
* storage.buckets.get

Result: PASS

**Fail**

Custom Role: custom.readonly.support

Permissions:

* compute.instances.get
* compute.instances.list
* compute.instances.start

Result: FAIL

Reason:
The permission `compute.instances.start` modifies a resource and is not considered read-only.

### Failure Condition

A read-only group contains one or more permissions that allow resource creation, modification, deletion, privilege escalation, or administrative actions.

### Alert

Email notification sent to IAM Operations team including:

* Group Name
* Assigned Role
* Permission(s) causing the violation
* Resource Scope
